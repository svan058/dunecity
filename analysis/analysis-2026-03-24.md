## 3. What from Micropolis Should Be Integrated

### 3.1 Core Simulation Systems (C++ to C++)

|| System | Source Location | Integration Approach |
|--------|-----------------|---------------------|
| **Zone Growth** | `micropolis/micropolis-activity/src/sim/zone.tcl` OR port from `zone.cpp` | Convert to C++ |
| **Infrastructure (Power)** | `MicropolisCore/MicropolisEngine/src/power.cpp` | Flood-fill algorithm |
| **Traffic Generation** | `MicropolisCore/MicropolisEngine/src/traffic.cpp` | Pathfinding for zones |
| **Census/Tax** | `MicropolisCore/MicropolisEngine/src/budget.cpp`, `evaluate.cpp` | Economy layer |
| **Disasters** | `MicropolisCore/MicropolisEngine/src/disasters.cpp` | Later phase |
| **Map Layers** | `MicropolisCore/MicropolisEngine/src/map.cpp` | Multi-res overlays |
| **16-Phase Scan** | `MicropolisCore/MicropolisEngine/src/scan.cpp`, `simulate.cpp` | Tick integration |

**Note:** The classic C++ MicropolisCore codebase (22,649 LOC) is the primary source. Key files:
- `micropolis.h/cpp` — Main engine class
- `zone.cpp` — Zone growth logic
- `power.cpp` — Power grid flood-fill
- `scan.cpp` — 16-phase simulation
- `budget.cpp` — Tax and budgets
- `traffic.cpp` — Road networks

### 3.2 Key Algorithms to Extract

1. **16-Phase Scan** (`simulate.cpp` lines 200-400)
   - Distribute zone updates across 16 phases
   - Prevents frame spikes

2. **Power Flood-Fill** (`power.cpp`)
   - `floodFillPower()` — spreads power from plants through conductive tiles

3. **Zone Desirability** (`zone.cpp`)
   - `getDesirability()` — composite of land value, pollution, crime, traffic

4. **Growth Valves** (`evaluate.cpp`)
   - `resValve`, `comValve`, `indValve` — balance controls

### 3.3 Data Structures

The `Map<DATA, BLKSIZE>` template pattern for multi-resolution layers is valuable:
```
Map<uint16_t, 1> — 1:1 tile map
Map<uint8_t, 2>  — 2:1 power grid, traffic density
Map<uint8_t, 4>  — 4:1 pollution, crime, land value
Map<int8_t, 8>   — 8:1 rate-of-growth
```

## 6. Unit Testing Strategy

### 6.1 Recommended Framework: gut

Godot's native testing is minimal. **gut** (Godot Unity Test) provides:
- Assertions (`assert_eq()`, `assert_true()`, etc.)
- Test doubles/mocking
- Parameterized tests

Install: Copy `gut.gd` to `addons/gut/` or use Godot Asset Library.

### 6.2 Test Categories

#### A. Determinism Tests (CRITICAL)

```gdscript
# tests/test_determinism.gd
func test_rng_reproducibility():
    var rng1 = SimRNG.new()
    var rng2 = SimRNG.new()
    rng1.set_seed(12345)
    rng2.set_seed(12345)
    
    for i in range(100):
        assert_eq(rng1.randi(), rng2.randi(), "RNG should be deterministic")
```

## 7. Migration Path from C++ to Godot

### 7.1 Three-Phase Approach

```
Phase 1: GDScript Prototype (We are here)
    ↓
Phase 2: Hybrid GDScript + GDExtension (Performance-critical paths)
    ↓
Phase 3: Full GDExtension (If needed)
```

### 7.2 Phase 1: GDScript (Current)

**What's done:**
- Game loop (`game_state.gd`)
- Entity base classes
- RNG (`sim_rng.gd`)

**What's needed (next 2-4 weeks):**
1. `Map.gd` — Tile grid, terrain, spice
2. `Pathfinding.gd` — A* in GDScript (slow but works)
3. Basic entities (Harvester, WindTrap, Refinery)
4. `gut` testing framework

**Performance expectation:**
- 50-100 units: OK in GDScript
- 200+ units: Need GDExtension

### 7.3 Phase 2: Hybrid (Weeks 4-8)

Identify hot paths via profiling:
1. Run game with 200+ units
2. Profile in Godot
3. Migrate slowest functions to C++ GDExtension

**Candidates for GDExtension:**
- `AStarSearch` — Most expensive
- `SpatialGrid` queries — O(n) per query
- Entity `tick()` for 100+ units
- Power flood-fill on large maps

**GDExtension structure:**
```
dunecity-godot/
├── addons/
│   └── dunecity_core/
│       ├── native/
│       │   ├── CMakeLists.txt
│       │   ├── src/
│       │   │   ├── pathfinding.cpp
│       │   │   ├── spatial_grid.cpp
│       │   │   └── simulation.cpp
│       │   └── register_types.cpp
│       └── bin/
│           └── libdunecity_core.so (or .dll)
└── scripts/
    └── sim/
        └── (GDScript wrappers)
```

### 7.4 Phase 3: Full C++ (If Needed)

Only if:
- GDScript + GDExtension hybrid doesn't meet performance
- Need to reuse more Dune Legacy code directly

**Complexity:** High. Requires:
- Full build system for C++ in Godot
- Complex IPC between GDScript and C++
- More maintenance burden

**Recommendation:** Avoid unless proven necessary.

### 7.5 Code Porting Examples

#### C++ → GDScript Example

**Dune Legacy (C++):**
```cpp
// src/Map.cpp (simplified)
class Map {
    std::vector<Tile> tiles;
    int width, height;
    
    Tile& getTile(int x, int y) {
        return tiles[y * width + x];
    }
    
    bool isSpice(int x, int y) {
        return getTile(x, y).terrain == Terrain::SPICE;
    }
};
```

**Godot (GDScript):**
```gdscript
# scripts/sim/map.gd
class_name Map

var tiles: PackedInt32Array  # Flat array, y * width + x
var width: int
var height: int

enum Terrain { SAND, ROCK, DUNE, SPICE, THICK_SPICE, MOUNTAIN }

func get_tile(x: int, y: int) -> int:
    return tiles[y * width + x]

func is_spice(x: int, y: int) -> bool:
    return get_tile(x, y) == Terrain.SPICE
```

#### Micropolis → GDScript Example

**Micropolis (Tcl):**
```tcl
# micropolis-activity/src/sim/power.tcl
proc floodFillPower {x y} {
    set visited [dict create]
    set queue [list [list $x $y]]
    
    while {[llength $queue] > 0} {
        set current [lindex $queue 0]
        set queue [lrange $queue 1 end]
        # ... process tile, add neighbors
    }
}
```

**Godot (GDScript):**
```gdscript
# scripts/sim/power_grid.gd
class_name PowerGrid

func flood_fill(start_x: int, start_y: int) -> Array[Vector2i]:
    var visited: Dictionary = {}
    var queue: Array[Vector2i] = [Vector2i(start_x, start_y)]
    var connected: Array[Vector2i] = []
    
    while queue.size() > 0:
        var current = queue.pop_front()
        var key = "%d,%d" % [current.x, current.y]
        if visited.has(key):
            continue
        visited[key] = true
        connected.append(current)
        # Add neighbors to queue...
    return connected
```

## 9. Summary

**Dune Legacy ports needed:** Map, Game Loop, Pathfinding, Object Manager, Entities (Units/Structures), House/Factions, Command Pattern

**Testing:** Install `gut` immediately. Write determinism tests before anything else.

**Migration path:** GDScript now → profile → migrate hot paths to GDExtension → full C++ only if proven necessary.
